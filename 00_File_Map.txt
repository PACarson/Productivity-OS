/**

 *      仅做 legacy 字符串转换，不直接调 RecurringEngine 的续期逻辑)
 *   → 09_TemporalParser.gs  (extractDateTime)
 *   → 05_SheetUtils.gs  (_cleanTitle_，纯文本工具，不是 Sheet 读取)

 * 09_IdempotencyManager.gs
 *   → 07_IdentityEngine.gs  (generateTaskIdentity)
 *   → 08_DeduplicationEngine.gs  (findExistingTask)
 *   → 20_TaskEngine.gs  (TaskEngine.createTaskDirect_)

 *
 * 08_DeduplicationEngine.gs → 05_SheetUtils.gs (getSheet_, getHeaderMap_)
 * 07_IdentityEngine.gs ← 零外部依赖
 *
 * 10_ProjectionEngine.gs
 *   → 05_SheetUtils.gs  (upsertRowByKey_, deleteRowByKey_, getSheet_, getHeaderMap_)

 * 11_ProjectionRebuilder.gs
 *   → 02_EventBus.gs (getAllEvents)

 *   → 05_SheetUtils.gs (getSheet_, getHeaderMap_, upsertRowByKey_, batchUpsertRowsByKey_)
 *   → 07_IdentityEngine.gs (generateTaskIdentity)
 *   → 20_TaskEngine.gs (deriveFromEvent)
 *
 * 12_TaskQueryEngine.gs
 *   → 05_SheetUtils.gs (getSheet_, getHeaderMap_)
 *   → 24_ViewEngine.gs, 22_PriorityEngine.gs, 23_SearchEngine.gs,
 *     25_DashboardEngine.gs, 26_AnalyticsEngine.gs
 *   （TaskQueryEngine 是这五个纯函数 Engine 唯一的调用方之一——
 *    05_DashboardEngine 内部也会调 24/26，但不会自己读 Sheet）
 *
 * 13_ActiveTasksEngine.gs → 05_SheetUtils.gs (getSheet_, getHeaderMap_)
 *
 * 15_Setup.gs → 13_ActiveTasksEngine.gs (triggerDailyArchive，createTriggers挂载用)
 *
 * 20_TaskEngine.gs
 *   → 09_IdempotencyManager.gs (createTaskIfNotExists)
 *   → 12_TaskQueryEngine.gs (getTask，completeTask取快照用)
 *   → 21_RecurringEngine.gs (spawnNextIfNeeded)
 *   → 02_EventBus.gs (publish, getAllEvents)
 *   → 05_SheetUtils.gs (upsertRowByKey_, deleteRowByKey_, shallowCopy_)
 *
 * 21_RecurringEngine.gs
 *   → 09_TemporalParser.gs (computeNextDueDateFromLabel)
 *   → 09_IdempotencyManager.gs (createTaskIfNotExists，生成下一次实例用)
 *
 * 22_PriorityEngine.gs ← 纯函数，零 Sheet/Event 依赖
 * 23_SearchEngine.gs   ← 纯函数，零 Sheet/Event 依赖
 * 24_ViewEngine.gs     ← 纯函数，零 Sheet/Event 依赖
 *
 * 25_DashboardEngine.gs
 *   → 24_ViewEngine.gs (今日/本周/逾期等视图过滤)
 *   → 26_AnalyticsEngine.gs (统计数字)
 *
 * 26_AnalyticsEngine.gs
 *   → 05_SheetUtils.gs (round1_/round2_，纯计算工具)
 *   → 02_EventBus.gs (仅 replayCompletionTrend_ 例外调用 getAllEvents，
 *      见文件头注释)
 *
 * ── 架构铁律（V4，见 00_Project_Constitution.gs P6 完整版本）─────────────
 * 1. 箭头只能往更底层指，禁止循环依赖；本项目不依赖 Core / Reminder OS
 *    的任何代码
 * 2. Events 表（共享）里 TASK_ 开头和 REMINDER_SENT 类型的事件只有本项目的
 *    02_EventBus.publish() 能写
 * 3. Tasks/ActiveTasks/TaskStatistics/TaskFilters（Read Model）只有本项目的
 *    10_ProjectionEngine 能写（materializeTaskRow_ 是安全兜底，属例外）；
 *    ArchiveTasks 只有 13_ActiveTasksEngine.runDailyArchive() 能写
 * 4. 所有查询必须经过 12_TaskQueryEngine，禁止直接 derive Events，
 *    禁止 View/Priority/Search/Dashboard/Analytics 五个纯函数 Engine
 *    自己读 Sheet
 * 5. 所有 Create 操作必须经过 09_IdempotencyManager，禁止直接
 *    EventBus.publish CREATE 事件；Update 操作经由 20_TaskEngine.updateTask
 *    直接发 TASK_UPDATED（不需要幂等去重，因为不涉及"是否已存在"判断，
 *    只是改已存在任务的字段）
 * 6. 只有 26_AnalyticsEngine.gs 明确标注的历史趋势函数可以重放 Events，
 *    其余任何模块（尤其是 Telegram 指令路径上的一切）禁止重放
 */

/**
 * 跟上面「二、模块关系」的区别：「二」画的是"谁调用谁"的具体函数级依赖图，
 * 本节画的是每个文件在 Clean Architecture 四层里属于哪一层——这是更粗粒度
 * 的分类，用来配合 00_Project_Constitution.gs「零之四、Dependency Rules」
 * 检查"有没有箭头往上指"。00_ 开头的四份治理文档（Constitution/State/
 * File Map/ADR）以及本次新增的 00_Roadmap.gs 不参与这个分层——它们属于
 * Blueprint 的 Governance 顶层分类，跟"请求怎么被处理"这件事无关。
 *
 * Architecture Layer Map
 *
 * Presentation
 *   06_TaskIntentParser.gs
 *     ↓
 * Application
 *   08_DeduplicationEngine.gs
 *   09_IdempotencyManager.gs
 *   11_ProjectionRebuilder.gs
 *   12_TaskQueryEngine.gs
 *   15_Setup.gs
 *   20_TaskEngine.gs
 *     ↓
 * Domain
 *   07_IdentityEngine.gs
 *   09_TemporalParser.gs
 *   21_RecurringEngine.gs
 *   22_PriorityEngine.gs
 *   23_SearchEngine.gs
 *   24_ViewEngine.gs
 *   25_DashboardEngine.gs
 *   26_AnalyticsEngine.gs
 *     ↓
 * Infrastructure
 *   01_SecureConfig.gs
 *   02_EventBus.gs
 *   03_Output.gs
 *   05_SheetUtils.gs
 *   10_ProjectionEngine.gs
 *   13_ActiveTasksEngine.gs
 *
 * 共 21 个文件，每个文件恰好出现一次（"所有文件必须归类到唯一 Layer"，
 * 见需求方 Upgrade 4 原文）。分类理由简述：
 *
 *   Presentation（1个）：唯一"知道 Telegram 是什么"的文件——负责把用户
 *   原始文字变成意图，把 Domain/Application 层返回的数据拼成 Telegram
 *   看得懂的文案。
 *
 *   Application（6个）：这几个文件都是"协调 Domain + Infrastructure 完成
 *   一个具体用例"，但自己不承载可复用的业务判断规则——09_IdempotencyManager
 *   协调锁+去重+创建；08_DeduplicationEngine 是"检查 Read Model 里是否已
 *   存在"这个应用层判断（直接读 Sheet，因此不能算 Domain，Domain 层规定
 *   不摸 Sheet）；12_TaskQueryEngine 协调"读 Sheet 一次 + 分发给 5 个
 *   Domain Engine"；20_TaskEngine 协调"发 Event + 调用 Recurring"完成
 *   create/update/complete/cancel 四个用例；11_ProjectionRebuilder 和
 *   15_Setup 协调"重放 Events/迁移 Schema"这类管理用例（这两个文件在
 *   Blueprint 宏观分类里属于 Operations，跟这里的 Application 微观分层
 *   是两套不同的问题，见 00_Project_Constitution.gs 零之四开头的说明）。
 *
 *   Domain（8个）：承载具体、可复用的业务规则，全部不摸 Sheet/Events（
 *   09_TemporalParser 例外允许调用 Utilities/Session 这类 GAS 内建纯函数
 *   API，不算 Infrastructure I/O）。22/23/24/25/26 五个是纯函数 Engine
 *   （Architecture Principles 第2条）；07_IdentityEngine 是纯哈希计算；
 *   21_RecurringEngine 基本纯函数，唯一的例外依赖见下方「已知例外」。
 *
 *   Infrastructure（6个）：直接做 I/O 或提供不含业务判断的机械工具——
 *   02_EventBus（Events 表读写）、05_SheetUtils（Sheet 读写基础设施）、
 *   10_ProjectionEngine（Read Model 写入）、01_SecureConfig（配置读取）、
 *   03_Output（尚未启用的输出工具）、13_ActiveTasksEngine（机械的"按
 *   retention 天数搬行"，不含打分/排序等业务判断，故不归 Domain）。
 *
 * 已知例外（跟 00_Project_Constitution.gs 零之四「已知例外」是同一件事，
 * 这里只是从"文件归类"这个角度再确认一遍，不是第二个例外）：
 *   21_RecurringEngine.gs（Domain）→ 09_IdempotencyManager.gs（Application）
 */
